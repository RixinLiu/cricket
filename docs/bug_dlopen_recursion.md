# Bug Report: Stack Overflow via Infinite `dlopen` Recursion in cricket-client.so

**Date:** 2026-03-03
**Branch:** start-over
**Status:** Fixed
**File changed:** `cricket/cpu/cpu-libwrap.c`

---

## Symptom

`run_cricket_eager.sh` crashed on every run with a segmentation fault shortly after
vLLM printed `"Using max model len 512"`. No useful error message was produced —
the process died silently. The shell script reported exit code 139 (SIGSEGV).

```
INFO 03-03 17:01:15 [model.py:1510] Using max model len 512
./run_cricket_eager.sh: line 50: 201581 Segmentation fault (core dumped) (...)
ERROR: Benchmark failed.
```

---

## Diagnosis

### Step 1: Read `journalctl`

No core dump was available (apport intercepts them). Reading kernel logs via `journalctl`
revealed the crash details:

```
python3[201581]: segfault at 7ffda89e7ff8 ip 000073073dc4a254
                 sp 00007ffda89e8000 error 6 in ld-linux-x86-64.so.2
```

Key observations:
- **crash location:** `ld-linux-x86-64.so.2` (the dynamic linker), not CUDA code
- **fault address:** `0x7ffda89e7ff8` = `sp - 8` (exactly 8 bytes below the stack pointer)
- **error 6:** write fault to an unmapped page (the stack guard page)
- **conclusion:** this is a **stack overflow**, not a bad pointer dereference

### Step 2: Identify the recursion

The crash happens inside the dynamic linker during library loading/symbol resolution.
The only code path that repeatedly invokes the dynamic linker from within cricket is
`dlopen()`. Tracing the call chain through two files:

**`cpu/cpu-libwrap.c` — `libwrap_get_sohandle()`** is called by every CUDA stub
generated by the `DEF_FN` macro in `cpu-libwrap.h`. Its job is to lazily open the
underlying CUDA library so `dlsym` can look up real function pointers. It calls
the plain libc `dlopen`:

```c
static const char* LIBCUDA_PATH = "/usr/local/cuda/lib64/libcudart.so";
static void *so_handle;

inline void* libwrap_get_sohandle() {
    if (!so_handle) {
        so_handle = dlopen(LIBCUDA_PATH, RTLD_LAZY);  // BUG: calls our own interceptor
    }
    return so_handle;
}
```

**`cpu/cpu-client.c` — cricket's `dlopen` interceptor** intercepts every `dlopen`
call in the process. It has a list of library paths to replace with `cricket-client.so`
itself. That list includes the exact path used by `libwrap_get_sohandle()`:

```c
static const char *replace_libs[] = {
    "libcuda.so.1",
    "libcuda.so",
    "libnvidia-ml.so.1",
    "libcudnn_cnn_infer.so.8",
    "/usr/lib/x86_64-linux-gnu/libcuda.so.550.127.05",
    "/usr/local/cuda/lib64/libcudart.so"   // ← matches libwrap's LIBCUDA_PATH exactly
};

void *dlopen(const char *filename, int flag) {
    ...
    for (size_t i = 0; i != replace_libs_sz; ++i) {
        if (strcmp(filename, replace_libs[i]) == 0) {
            dl_handle = dlopen_orig("cricket-client.so", flag);  // trigger ld-linux
            return dl_handle;
        }
    }
    ...
}
```

### Step 3: The infinite recursion cycle

```
vLLM calls first CUDA API (at CUDA context init time)
  └─ cricket DEF_FN stub (e.g. cudaGetDeviceCount)
       └─ libwrap_get_sohandle()                        [cpu-libwrap.c]
            └─ dlopen("/usr/local/cuda/lib64/libcudart.so")
                 └─ cricket dlopen interceptor fires    [cpu-client.c]
                      └─ dlopen_orig("cricket-client.so")
                           └─ ld-linux resolves symbols in cricket-client.so
                                └─ CUDA stub constructors / initializers run
                                     └─ libwrap_get_sohandle()
                                          └─ dlopen("/usr/local/cuda/lib64/libcudart.so")
                                               └─ cricket dlopen interceptor fires
                                                    └─ ... (repeats forever)
```

Each recursion level consumes stack space inside `ld-linux`'s symbol resolution
routines. After enough levels the 8 MB stack is exhausted and the guard page write
triggers SIGSEGV.

The same recursion applies to `libwrap_get_nvml_handle()`, which opens
`"libnvidia-ml.so.1"` — also in the intercept list.

### Why it was hard to catch

- No error message: the process died inside the dynamic linker before any user code ran
- No core dump: apport intercepted it
- gdb approach was too slow: vLLM takes ~60s to load before hitting the crash point
- The `libwrap_get_sohandle()` guard (`if (!so_handle)`) gives a false sense of safety —
  it does NOT protect against recursion because `so_handle` is still NULL on every
  re-entrant call (it hasn't been set yet when the recursion starts)

---

## Root Cause

`libwrap_get_sohandle()` called the process-wide `dlopen()`, which cricket had
already replaced with its own interceptor. The interceptor did not distinguish between
external callers (the application) and internal callers (cricket itself), so it
treated cricket's own internal library-open as an application request to redirect
CUDA calls — and redirected it right back into cricket.

---

## Fix

**File:** [cricket/cpu/cpu-libwrap.c](../../../cricket/cpu/cpu-libwrap.c)

Use `dlsym(RTLD_NEXT, "dlopen")` to obtain the real glibc `dlopen` — the one that
exists *after* cricket-client.so in the dynamic linker's search order — and use it
exclusively for cricket's internal library loading. This completely bypasses the
interceptor.

```c
// Before (buggy):
inline void* libwrap_get_sohandle() {
    if (!so_handle)
        so_handle = dlopen(LIBCUDA_PATH, RTLD_LAZY);  // intercepted → infinite recursion
    return so_handle;
}

// After (fixed):
#define _GNU_SOURCE   // required for RTLD_NEXT
#include <dlfcn.h>

static void *(*real_dlopen)(const char *, int) = NULL;

static void *libwrap_dlopen(const char *path, int flags)
{
    if (real_dlopen == NULL)
        real_dlopen = dlsym(RTLD_NEXT, "dlopen");  // get real glibc dlopen
    return real_dlopen(path, flags);
}

inline void* libwrap_get_sohandle() {
    if (!so_handle)
        so_handle = libwrap_dlopen(LIBCUDA_PATH, RTLD_LAZY);  // bypasses interceptor
    return so_handle;
}
```

Same fix applied to `libwrap_get_nvml_handle()`.

`RTLD_NEXT` tells `dlsym` to start searching from the library *after* the caller
(cricket-client.so) in the link map, which resolves to glibc's real `dlopen`.
Cricket's interceptor is never invoked.

## General Lesson

When a library intercepts `dlopen` at the process level (via `LD_PRELOAD`), any
internal use of `dlopen` within that library **must** call the real underlying
`dlopen` directly, not the process-wide symbol. The standard idiom is:

```c
static void *(*real_dlopen)(const char *, int) = NULL;
// call once at init or lazily:
real_dlopen = dlsym(RTLD_NEXT, "dlopen");
```

Failure to do so creates a self-referential loop whenever the intercepted path
appears in the library's own initialization code.
